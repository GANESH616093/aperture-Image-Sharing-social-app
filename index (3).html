How a photo travels

Six managed services, one upload. Nothing here is a guess — it's the same shape a small photo-sharing app would run in production.

01 — INTAKE

The front desk

Amazon API Gateway

Every request from the app — upload, like, comment — arrives here first. It checks the request is well-formed and hands it to the right function.

02 — PROCESSING

The darkroom techs

AWS Lambda

Short-lived functions do the actual work: writing the file to storage, kicking off moderation, and updating records — no server sitting idle between uploads.

03 — INSPECTION

The inspector

Amazon Rekognition

Scans each image as it lands: auto-generates tags like "Outdoor" or "Portrait" and flags anything that shouldn't reach the feed.

04 — ARCHIVE

The negatives locker

Amazon S3

The original file lives here, durably, once and only once. Everything else in the system points back to this copy rather than duplicating it.

05 — DELIVERY

The gallery wall

Amazon CloudFront

Caches images at edge locations close to whoever's viewing, so a photo opens just as fast in another city as it does next door.

06 — LEDGER

The guestbook

Amazon DynamoDB

Holds the facts about each photo — captions, tags, like counts, comment threads — and answers feed queries in single-digit milliseconds.

Upload tap→ API Gateway→ Lambda→ S3 (store) + Rekognition (scan)→ DynamoDB (record)→ CloudFront (serve)

About this build

This is a front-end demo of the product experience. Uploads, likes, and comments run entirely in your browser (saved to local storage), and the AWS pipeline above is simulated on upload so you can see the shape of each step.
